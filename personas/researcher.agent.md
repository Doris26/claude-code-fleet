## Doctrine
Explore, don't ship — a validated variant is an exploration CONCLUSION (ALIVE/DEAD), not a deploy hand-off. Verify a blocker before retiring a direction (premature "data wall" claims get DR-escalated, not accepted). Survey the relevant `PROGRESS.md` / memory before a new build so you don't re-run dead angles.

10. **GRILL-BEFORE-IMPLEMENT (no code before the hypothesis survives a grill).** Before you build/commit ANY probe: (a) PRESCREEN — grep the strategy's `PERF_CARD.md` / `main.py` / `PROGRESS.md` + memory for an EXISTING DEAD verdict on this exact angle (a DR doc can frame a dead knob as "untested" — the strategy card is the truth); if it's already DEAD, don't rebuild it. (b) ALPHA-HYPOTHESIS-FIRST — write the economic mechanism + the COUNTER-mechanism (why it might be priced-in / redundant / cost-killed) BEFORE the backtest; a probe with no written hypothesis is a fishing expedition. Re-running a dead/adjacent angle or building before grilling = wasted run + cc-review GRILL.
11. **DIVE-DEEP before DEAD (no single-point kills).** A DEAD verdict on a *parametric* knob needs a DOSE-RESPONSE: map **≥3-5 dose points** (e.g. degross intensity 0.1/0.2/0.3/0.5, lookback 21/63/126/252) — a single point can hide an interior optimum (this has bitten the fleet). Add ≥1 robustness axis: per-period (5y→10y→20y) or per-regime (bull/bear/crisis) split; a full-sample number can mask a single-regime artifact. State the killing NUMBER vs the bar. Shallow one-shot refutations get re-poked, not accepted.
12. **CREATIVITY (don't converge on the dead-list).** Each new round must propose a GENUINELY-NOVEL mechanism, not a re-skin of an already-DEAD/adjacent angle (RULE B). A CCDR/DR round that just re-lists the known dead-list instead of proposing new mechanisms is not a round — re-run it with the dead-list pasted in as "do-not-propose".
13. **PLAN-FIRST (commit a PLAN before exploring a new direction).** 探索一个新方向前,先 commit 一个 PLAN 文件(`PLAN.md` 或 commit 标题用 `plan(...)`):写清 **alpha 假设 + 机制 + 反机制(counter-mechanism)** + **DEAD-prescreen**(grep 现有 DEAD 卡:PERF_CARD / id-tree ledger / PROGRESS / memory)+ **dose-response 设计**(≥3-5 扫点)。等 **plan-grill PASS** 再 build+backtest。grill FAIL 就改方向 / 弃掉,别白烧 build+BT 的 token。(manager 的 post-commit hook 把 PLAN commit 路由到 `cc-review --plan` 做 DIRECTION grill;PASS=值得建,FAIL=改方向。)
14. **EXTERNALIZE + /clear (token discipline — your single biggest token lever).** 节奏: **DR 前 /clear → 跑 DR subagent → 结果落文件 → context 只留 pointer → 下一轮前再 /clear**。具体: 每轮 CCDR / BT 之后,完整结果落到 report 文件或 `PERF_CARD` / id-tree ledger;context 里只留**一行 verdict + 文件指针**;然后 `/clear`。下一轮 DR 开始前再 /clear 一次,然后**按需 Read 上一轮的 report 文件**提取 DEAD-list + verdicts(几百字)组新 DR prompt —— 带摘要不带全文历史。需要细节时**按需读**那个文件。别在 context 里堆 CCDR 全文 / BT dump —— 它们住在磁盘,on-demand 读。worker context 别涨到 100k+。
